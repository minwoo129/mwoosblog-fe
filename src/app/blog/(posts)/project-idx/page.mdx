# 1. Project IDX가 뭐야?

![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image02.webp)
Project IDX는 지난 구글에서 최초로 공개한 웹 브라우저 기반 IDE입니다. 우리가 보통 개발을 할 때는 각자가 개발하고자 하는 것들에 대한 환경설정을 컴퓨터에 적용하고 이를 통해 VSCode, IntelliJ와 같은 IDE를 사용해 코드를 작성하는 방식으로 진행합니다. 하지만 Project IDX는 컴퓨터 로컬 환경이 아닌 웹 브라우저를 통해 구글의 클라우드 환경에서 진행하는 방식인 것입니다. Project IDX는 Codey라는 PaLM2 모델 기반의 AI 코딩 도구를 통해서 동작하며 IDX를 통해서 개발된 프로젝트 중 일부 웹 프론트 프로젝트는 그 자리에서 바로 Firebase Hosting을 통해 배포가 가능합니다.
현재 Project IDX는 지난해 말에 공개된 이후 오픈 베타로만 운영되고 있는 상태입니다.

> **알려드립니다**  
> Project IDX는 현재 공식적으로 출시가 완료되었으며, 현재 Firebase 소관으로 이전되었습니다.  
> 따라서, 공식적인 명칭은 [Firebase Studio](https://firebase.studio/) 입니다.

# 2. 사용 가능한 개발 프레임워크

Project IDX는 리눅스 환경에서만 작업이 가능한 경우를 제외하고 VSCode나 IntelliJ와 같이 컴퓨터에 설치해서 사용하는 IDE를 사용하는 프레임워크인 경우에 사용이 가능한데, 아직 베타인데도 상당히 많은 개발 프레임워크를 지원하고 있습니다.

일단 웹 프론트에서 **Angular, NextJS, Astro, React, 기본 HTML, Svelte, Vue.js, Nuxt** 등의 프레임워크를 지원합니다.

백엔드는 **Go, Java, Dart, C++, PHP, Flask(Python), nodejs(express만 지원), Rust, Django(Python)** 등을 지원하며 Go와 Python을 사용하는 백엔드의 경우 **Gemini의 생성형 AI를 사용할 수 있는 API**도 지원합니다.

모바일은 주로 크로스 플랫폼 프레임워크 위주로 지원이 가능한데 일단 구글이 운영하는 **Flutter**는 기본적으로 지원하고 **React Native**는 **현재 개발단계**에 있는 상태이지만 시범적으로 사용이 가능했습니다. 또한 Android 네이티브 방식도 현재 **Android Studio IDE** 환경 개발을 개발하고 있고 React Native 처럼 시범적으로 사용이 가능했습니다.

AI와 머신러닝의 경우 **Gemini API**를 사용하는 방식과 **Firebase Genkit을 사용한 생성형 AI 모델 개발**이 가능하며 Gemini를 사용해 **LangChain**(LLM 기반의 앱을 구축하는 과정을 간소화할 수 있는 오픈소스 프레임워크)도 사용 가능했습니다.

데이터베이스 개발은 현재 **Postgres SQL**을 포함해 **Prisma, Firebase Data Connect** 등을 사용할 수 있으며 **MySQL**은 현재 개발단계의 상태를 시험해볼 수 있는 상태입니다.

# 3. 주요 특징

Project IDX의 첫번째 특징은 우선 Code OSS를 기반으로 개발되어 **VSCode와 UI가 거의 유사**하고 **사용방식도 VSCode와 거의 유사한** 것이 특징입니다.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image03.png)

그리고 구글의 Github Copilot이라고 할 수 있는 Gemini Code Assist을 자체 탑재하고 있고 이 기능은 별도로 설정하지 않아도 자동으로 활성화됩니다.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image04.png)
그리고 VSCode에서 Github Copilot을 사용해보신 분이라면 아시겠지만 GPT와 같이 채팅창의 형태로 질문할 수 있는 창이 있는 것도 VSCode와 동일했습니다.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image05.png)

그 다음으로는 모바일 개발을 하시는 분이라면 안드로이드, iOS 단말기의 에뮬레이터가 있어야 테스트가 가능한데 Android, 웹 환경에서의 에뮬레이터도 지원합니다. 그리고 iOS 시뮬레이터의 경우에는 현재는 지원하지 않지만 도입 예정이라고 하는 것을 보아 아직까지 개발단계에 있는 것으로 보여집니다.

그리고 그 외에 기본적으로 기존에 Github에 올라가있는 프로젝트를 그대로 클론해와서 작업도 가능하고 특히 Nextjs와 같이 SPA 기반 웹 프레임워크의 경우 Project IDX를 사용해서 바로 Firebase Hosting에 배포 가능하다고 합니다.

# 4. 사용후기(프레임워크 별)

제가 이번에 Project IDX를 실제로 사용을 해보고 이에 대한 간단한 후기를 정리해보려고 합니다. 프로젝트는 React Native, Flutter, NextJS, nodejs 등을 사용해서 모바일, 웹 프론트, 백엔드 등 다양한 방식에서 테스트를 진행하였습니다.

## 4-1. React Native

우선 React Native를 리뷰를 하자면, 일단 React Native는 CLI가 아니라 Expo를 사용하는 방식이며, 언어는 TypeScript만 지원됩니다. 보통 React Native는 Expo를 사용해서 개발하는 경우 안드로이드, iOS 네이티브 영역에 접근해 세부적인 설정을 해줘야 하는 경우에는 적합하지 않기 때문에 실제로는 CLI를 사용해 많이 개발을 하는 현장 상황을 고려하면... 사용하기에 적합하지는 않다는 생각을 가졌습니다. 그리고 아직 개발단계에 있어서 개발환경 가상머신을 실행하는 단계와 에뮬레이터 실행 단계가 상당히 불안정한 부분이 있었습니다. 이게 처음 프로젝트를 생성할 때는 괜찮았는데 그 다음에 실행할 때 거의 실행이 안된다 할 정도로 불안정했어요.
그리고 프로젝트의 파일 구조와 초기 코드 구조를 살펴봤을 때 기존에 Expo로 생성한 프로젝트와 이상할 정도로 차이가 커서 이게 잘못 빌드된건가 하고 약간의 의심을 가졌던 부분이 있습니다.

(아래 이미지: 기존에 컴퓨터에서 생성했었던 Expo 프로젝트)
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image06.png)

(아래 이미지: Project IDX로 생성한 Expo 프로젝트)
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image07.png)

다만 React Native는 아직 개발 단계에 있는 상태임에 따라 추후 코드 내용이 변경될 가능성도 있다고 보여집니다.

## 4-2. Flutter

그 다음으로는 Flutter입니다.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image08.png)

Flutter의 경우에는 iOS 시뮬레이터를 실행할 수 없다는 것 빼고는 기존에 VSCode에서 개발할 때와 거의 차이가 없었습니다. 그리고 아무래도 Flutter도 구글이 개발한 거다보니 React Native에 비해서는 가상머신 최적화가 잘 되어 있는 것 같다는 느낌을 받았습니다. 그리고 VSCode에서 개발할 때는 Flutter 디버그 환경을 위한 익스텐션을 깔아줘야 VSCode가 Flutter 디버그 실행을 위한 UI들이 표시됬는데, IDX에서는 이미 그 익스텐션이 설치된 상태로 프로젝트가 실행되서 더 이상 또 설정할게 없었어요. 터미널에서 기본적으로 사용하게 될 dart나 flutter CLI도 설치되어 있었고요. 그리고 신기했던 것이 Flutter 개발을 하시는 분이라면 아시겠지만 Flutter에서는 flutterfire라는 CLI 플러그인을 설치해서 이 CLI 명령어를 통해 Firebase에 바로 연결할 수 있습니다. 근데 이 플러그인도 이미 설치가 되어 있었어요. 그러니까 firebase에 연결하려면 터미널에서 firebase 로그인, 프로젝트 연결만 해주면 끝나는 거죠.

## 4-3. NextJS

그 다음은 NextJS 입니다.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image09.png)

NextJS도 VSCode에서 개발할 때와 기본적인 파일 구조와 코드 구조에서 큰 차이점을 느끼진 못했어요. 그리고 웹 프론트라서 그런지 모바일 때보다 조금 더 쾌적한 느낌을 받았습니다. 다만 한가지 신기했던 점은, 처음 프로젝트 생성하는 단계에서 원하는 언어 템플릿도 설정할 수 있었고, src/dir 폴더 구조 생성 여부, ESlint 활성화 여부, App Router 사용 여부, Tailwind CSS 기본 활성화 여부 등을 선택해서 생성이 가능했습니다.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image10.png)

## 4-4. nodejs(express)

마지막은 백엔드 개발을 테스트해봤고, nodejs를 사용하여 진행하였습니다.
nodejs도 NextJS와 같이 프로젝트 초기 생성 시 원하는 옵션을 선택해서 생성이 가능했습니다. 옵션은 좀 적었지만 API Server 템플릿과, 그냥 아무것도 없는 템플릿, TypeScript 사용설정 여부 등을 선택해 생성할 수 있었습니다.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image11.png)

저는 API Server 템플릿에 TypeScript를 활성화하여 프로젝트를 생성하였는데요.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image12.png)
express를 사용한 기본 API가 추가되어 설정되었습니다.
실제로 API 호출 테스트를 진행해본 결과는, 현장에서 사용하기에는 아직까지는 적합하지 않겠다라는 생각이 들었습니다. 보통 API 테스트를 하려면 Postman을 사용해서 API 테스트를 진행하는데, 크롬에서 호출할 때는 문제가 없었지만(Get 타입 호출만) Postman을 사용할 때는 401에러가 걸리면서 추가적인 설정이 필요했습니다. 이 내용들이 구글에 좀 올라오긴 했지만 아직까지 구글에서도 Postman 연결 절차에 대한 공식적인 방법을 제공하지 않았기 때문에 이에 대한 혼란은 좀 있을 거 같습니다.

그리고 백엔드 개발을 하게되면 보통 개발한 프로젝트를 Docker 컨테이너로 빌드해서 이를 웹서버로 띄우게 되는데, 이는 백엔드 프로젝트에 한해 프로젝트 내 설정 파일에서 Docker를 활성화하는 명령어 한줄만 추가해주면 별도의 설정 없이 터미널에서 Docker를 활성화할 수 있습니다. 이는 nodejs 뿐만 아니라 모든 백엔드 프로젝트에 동일하게 적용됩니다.

# 5. 전반적인 사용 후 느낀 점

그 다음은 위 4개의 프로젝트를 실행해보면서 공통적으로 느낀 사용 후기와 이에 더해 기대되는 점과 아쉬웠던 점들을 정리하고자 합니다.

## 5-1. VSCode 대비 살짝 느린 반응속도

우선 이 부분은 웹 브라우저에서 실행되는 특징 상 어쩔 수 없는 부분이겠지만 기존 VSCode 대비 반응속도가 좀 느려보였습니다. 지금 이 부분에 대해서는 구글에서도 인지를 하는 것으로 보여지는데요, 앞으로의 개발 방향성을 확인해본 결과 AI 워크로드 성능 향상을 위한 GPU 적용을 준비하고 있는 것 같습니다. 따라서 이 부분이 실제로 적용된다면 사용자의 인터넷 환경에 따라 차이가 있을 수 있지만 지금보다는 나아지지 않을까 생각이 듭니다.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/project-idx/image13.png)

## 5-2. iOS 시뮬레이터 미지원

그 다음으로는 모바일 프로젝트 개발시 iOS 시뮬레이터가 지원되지 않는 점이 많이 아쉬웠습니다. 사실 구글에 입장에서 크로스 플랫폼 프레임워크인 Flutter를 운영하면서 iOS 테스트를 위해서라도 iOS 시뮬레이터는 언젠가는 반드시 지원이 이루어지리라 생각합니다.
다만, 이 내용은 좀 조심스러운 생각이지만, iOS 앱 빌드 및 배포환경이 갖춰질 수 있을까라는 생각이 들었습니다. 구글에서 IDX를 개발하면서 가져간 비전은 코드를 작성해서 시뮬레이션에서 끝나지 않고 원스톱 배포까지 가능한 환경을 추구하였는데 모바일 프로젝트에서 안드로이드는 빌드 및 배포가 가능하지만 iOS는 그러지 못한다면 반쪽자리 환경으로 여겨질 수 있기 때문입니다. 만약 iOS 앱 빌드와 배포기능까지 현실화된다면 개발자 입장에서는 상대적으로 금액이 비싼 맥을 꼭 구매해야 하는 필요성이 줄어들고 윈도우에서도 iOS 개발의 모든 과정을 다 진행할 수 있게 되면서 정말 폭발적인 반응이 나타날 수 있다고 생각합니다.
하지만 이 기능의 경우에는 애플의 협조와 개발 협력이 필요한 사안인데, 만약 애플에서 맥 라인업, 그 중에서 가격대가 비싼 맥북프로, 맥스튜디오와 같은 고사양 맥의 주 소비층 중 하나인 개발자 층의 구매율 하락으로 이어질 것을 우려해 구글의 협력 요청을 거절한다면, iOS 앱 빌드, 배포 기능은 적용되지 않을 가능성도 있다고 보여집니다.

## 5-3. 일부 개발자 층의 제한적인 유입 가능성

다음으로 제가 느낀 점은 일부 개발자 층을 중심으로 IDX로의 유입이 제한적일 수 있다고 생각하였습니다. 특히 Java 또는 Python 기반의 백엔드 개발의 경우 Jetbrains 패밀리에 속하는 IntelliJ와 Pycharm을 사용하는 경우가 많은데 Jetbrains 패밀리 IDE와 IDX의 UI 및 세부기능이 완전히 다르기 때문에 이 부분이 보완되지 않는다면 백엔드 개발자 유입이 어렵다고 생각하였습니다.
그리고 React Native의 경우에도 지금은 Expo만 지원하지만 나중에 정식 출시됬을 때도 Expo만 지원하고 CLI는 지원하지 않는다면 CLI를 많이 사용하는 현장 특성상 React Native 개발자 층의 유입은 제한될 수 있다고 생각하였습니다.
특히 IDX는 (뭐 얼핏보면 당연한 거 겠지만) 구글의 개발 프레임워크를 적용했을 때 훨씬 안정성이 뛰어난 부분들이 있는거 같아서 이 경우가 아닌 이상 개발자들의 유입이 제한될 가능성이 있어보입니다.

## 5-4. AI 기능 사용

다음으로 IDX에서 제공하는 Gemini Code Assist와 관련해서는 이게 Github Copilot을 사용해보셨다면 아시겠지만 코드위에 수정 방법을 물어보는 부분과 채팅창 부분, 코드 입력 시 입력 코드를 추천해주는 기능이 Copilot이랑 거의 똑같습니다. 거의 98% 정도 비슷해보여요.
근데 이 기능은 저는 상황에 따라서 끌 수도 있어야 한다고 생각하는데 VSCode와 달리 IDX는 이 기능이 내장된 기능이라 이 기능을 끄는 방법을 찾지 못했습니다. 제가 못찾아서 그럴 수도 있고 진짜로 이 기능을 끄는 것이 불가능한 건지는 조금더 알아봐야 할거 같습니다.

# 마지막으로

마지막으로 아직까지는 IDX가 출시되지 않았기 때문에 언제 출시될지는 장담할 수 없습니다. 하지만 저는 개인적으로는 Project IDX의 방향과 발전 가능성에 대해 높게 평가하면서도 운영 후 방향에 대해서는 신중한 입장이기 때문에, 일단 정식 출시 이후에도 구글의 행보를 지켜봐야 할 것 같습니다. 그리고 IDX는 AI 기능을 포함해 단일 PC에서 개발할 때도 높은 성능의 컴퓨터를 필요로 할 정도로 높은 성능을 필요로 하는 기능들이 포함된만큼 무료로 출시하는 것은 절대로 불가능하기 때문에 정식 출시 이후 얼마 정도에 사용 요금이 책정될 지를 지켜봐야 할 거 같습니다.
그리고 무엇보다고 오래전부터 구글이 자체적으로 개발해왔고 멀쩡히 잘 돌아가고 있는 서비스들을 쥐도 새도 모르게 종료시킨 전례가 한둘이 아니기 때문에 IDX가 정식 출시되더라도 당장 사용하기 보다는 한 2~4년 정도 상황을 지켜보다가 넘어가는 것이 좋다고 생각이 듭니다.
작년에 공개된 이후 올해 구글IO에서는 IDX에 관해 추가적으로 발표된 내용은 없기 때문에, 내년 2025 구글IO에서 IDX와 관련한 추가적인 내용이 발표될지 지켜봐야 할 거 같습니다.
