오늘은 개발자라면 누구나 사용하고 있는 소스코드 변동사항 관리를 위한 "형상관리"에 대한 내용을 작성해보려고 합니다.(갑자기 생각이 나서 간단히 자료조사를 하고 글만 적었습니다 ㅎㅎㅎ)

# 1. 형상관리

형상관리는 영어약자로 SCM(Software Configuration Management)이라고 부르고, 소프트웨어 또는 소스코드의 변경사항을 체계적으로 끊임없이 관리하기 위한 개념입니다. 쉽게 설명하면 개발자가 소프트웨어 또는 소스코드를 작성하면서 발생하는 변경사항들을 다른 개발자와 협업하고, 경우에 따라 뒤로 되돌릴 수 있도록 체계적으로 관리할 수 있도록 하는 개념인 것입니다.
이러한 형상관리의 하위 개념으로는 버전관리시스템(VCS, Version Controll System), 소스코드관리(Source Code Management) 등 여러가지 개념이 존재하지만 통상적으로 형상관리라는 동일한 개념을 의미합니다.

# 2. SCCS(Source Code Controll System)

SCCS는 1972~1973년 벨 연구소에서 개발된 최초의 버전 관리 시스템(VCS)입니다. 파일 단위의 버전 관리를 제공하며 각 파일에 대한 수정내역을 추적할 수 있죠. 지금 우리가 많이 사용하는 Git과 원리는 거의 비슷합니다. SCCS는 파일의 각 버전을 '델타(delta, 수학에서는 변화량을 의미합니다)'라고 부르는 변경 내역으로 저장해 필요에 따라 특정 버전을 재구성하는 방식입니다. 하지만 SCCS는 다중 사용자 환경에서는 동시성 제어가 어려웠던 한계가 있어 이후에 나오는 RCS, CVS같은 툴에 밀려나게 됩니다.

# 3. RCS(Revision Controll System)

RCS는 1982년 출시된 버전 관리 시스템으로, 네트워크상이 아닌 로컬 상에서 관리하는 초기 형태의 시스템입니다. 파일 단위의 버전관리 기능을 제공한다는 점은 SCCS와 동일하지만 변경 내역을 순차적으로 추적하는 것이 주 목적이었습니다. 하지만 로컬 버전 관리시스템의 특성상 다중 사용자 환경에서의 협업이 제한된다는 점에서 주로 개인 사용자에 의해 많이 사용되었습니다. 시간이 지나면서 이보다 훨씬 더 고도화된 시스템이 나오면서 사용량이 줄어들긴 했지만 그래도 아직까지 사용하는 사례가 있다고 합니다.

# 4. CVS(Concurrent Versions System)

CVS는 1990년 출시된 형상관리 툴로, 기존 RCS에서 다중 사용자와의 협업에 있어서 한계를 극복하고자 출시된 **중앙 집중식 버전 관리 시스템**입니다. 중앙 저장소를 통해 파일을 관리하고, 사용자는 각자의 로컬 환경에서 파일을 수정해 이를 중앙 저장소에 커밋한 다음 merge 과정을 거쳐 변경사항을 병합합니다. 또한 RCS를 기반으로 개발되었지만, 파일 단위가 아닌 프로젝트 단위로 버전 관리를 제공한다는 점이 특징입니다. 현재 가장 많이 사용되고 있는 Git의 기본적인 개념과 비슷합니다.
다만, CVS도 merge 충돌을 해결하는 부분과 느린 속도 등이 한계로 지적되면서 이후 SVN(Subversion)으로 대체됩니다.

# 5. SVN(Subversion)

SVN은 CVS와 같은 중앙 집중식 버전 관리 시스템으로, 2000년 아파치(Apache)에 의해 개발 발표된 시스템입니다. SVN은 기존에 CVS에서 한계로 지적된 병합 충돌 처리에서의 어려움, 파일과 디렉토리의 이동 및 이름변경의 비효율성 등의 문제를 극복하기 위해 개발된 시스템으로, "**원자성**"이라는 개념 도입을 통해 데이터의 일관성을 유지할 수 있게 되었고, 이름 변경, 복사, 삭제 등의 기능을 지원하였습니다. 또한 클라이언트와 서버가 HTTP, HTTPS 등의 프로토콜로 통신할 수도 있었죠.(CVS는 일반적으로 SSH를 사용해 접근하는 방법을 사용했습니다.)
출시 이후 SVN에 의해 소프트웨어 개발의 효율성과 품질이 크게 향상되어 많은 개발자에게 신뢰할 수 있는 VCS로 자리잡아 VCS 발전에 중요한 역할을 하였습니다. 하지만 그럼에도 불구하고 여전히 중앙 서버가 존재해야 한다는 점과 코드의 병렬 개발 및 병합에 어려움이 존재했습니다.

# 6. Git

이후 SVN의 이러한 문제점을 개선한 새로운 시스템이 2005년 세상에 나오게 되는데요, 지금은 사용하지 않는 개발자가 없다고 해도 과언이 아닌 현대 형상관리 시스템의 제왕, 바로 Git입니다.
![](https://edqtpvyrloqd28014065.gcdn.ntruss.com/blog/vcs-review/image01.svg)

Git은 Linux를 개발해 세상에 공개한 오픈소스계의 아이돌, 리누스 토르발스가 개발해 2005년 출시한 분산형 버전 관리 시스템(VCS)입니다.

> **참고!!!**
> 개발자가 아닌 분도 어디서 들어보셨을 수 있지만, 이러한 Git 시스템을 제공하는 가장 대표적인 플랫폼이 바로 그 유명한 **"Github"**입니다.
> 그 외에도 Gitlab, AWS CodeCommit 등 여러가지 플랫폼이 존재하지만 전세계에서 가장 유명하고 압도적인 사용자 점유율을 차지하는 플랫폼이 바로 Github입니다.

기존 SVN이 중앙 집중식 VCS(CVCS, Centeralized Version Control System)인 것과 달리 Git은 분산형 VCS(DVCS, Distributed Version Controll System)로 중앙에 위치한 원격 저장소와 로컬 저정소 두 영역에서 모든 버전 관리 작업을 수행할 수 있어 SVN과 달리 네트워크 연결 없이도 모든 관리 작업을 수행할 수 있게 되었습니다.
또한 히스토리 관리에 있어서도 SVN은 한번 커밋을 하게되면 이를 수정하기가 매우 어려운 반면, Git은 상황에 따라 다르지만 커밋을 되돌리거나 하는 방법으로 자유롭게 수정, 변경할 수 있습니다.
그리고 Git은 속도, 데이터 무결성, Linux 커널과 같은 대규모 프로젝트를 지원하는 능력에 초점을 맞춰 개발되어 성능에 있어서도 SVN보다 월등한 것이 특징입니다.

지금은 현대 형상관리 시스템의 제왕으로 성장한 Git이지만, 이런 Git이 세상에 나오게 된데는 한가지 일화가 있다고 합니다. 리누스 토르발스가 처음에 Linux 커널 소스코드를 관리할 때는 아이러니(?)하게도 기존의 SVN이나 CVS 같은 VCS를 전혀 사용하지 않았다고 합니다. 이유는 단순히 **토르발스 본인이 마음에 들지 않았다**는 것이 정설입니다. 그러다가 결국 소스를 관리하는데 한계가 왔던 것인지 어쩔 수 없이 VCS를 사용하게 되는데, 당시에 토르발스가 선택한 툴은 바로 BitKeeper라는 VCS였습니다.

> 당시에 BitKeeper가 분산 처리 기능과 빠른 성능이 특징이지만 유료였기 때문에(지금은 무료입니다) 토르발스의 이런 선택은 자유 소프트웨어 진영으로부터 엄청난 비난을 받았지만 이 진영에서도 BitKeeper외에 다른 대안을 제시하지는 못했다고 합니다.

하지만 이후 오픈소스 생태계에 깊이 뿌리내려 정착한 Linux와 이윤을 추구하는 BitKeeper 사이에 갈등이 빚어지게 됩니다. 바로 BitKeeper 측에서 일부 Linux 개발자들의 리버스 엔지니어링을 문제삼아 해당 개발자들의 사용을 제한하는 일이 벌어진 것입니다. 그 일이 있고 토르발스가 (하도 빡친 나머지) 단 2주만에 개발해 세상에 공개한 VCS가 바로 Git이었다고 합니다.
그리고 Git이 세상에 공개되고 Git이 많은 개발자에게 대세로 자리잡으면서 뒤늦게 BitKeeper도 유료정책을 폐지하고 아파치 라이선스로 소스까지 공개했지만 Git이라는 새로운 대세를 꺾기에는 역부족이었습니다.
이를 통해 Git은 지금까지 세상에 나온 수많은 형상관리 툴들을 큰 격차로 누르고 압도적인 점유율의 형상관리 시스템으로 자리잡았습니다.

저는 이 글을 작성하면서 놀라웠던 점이, Git을 개발한 사람이 그 유명한 리누스 토르발스였다는 점과, Git이 세상에 나온지 생각보다 그렇게 오래되지 않았다는 점이 놀라웠습니다. 이제 Git이 세상에 나온지 올해(2025년)로 딱 20년이 되었네요!! (TMI, 저는 그 때 초등학교 2학년이었습니다 ㅋㅋㅋㅋ)

개발자들이라면 모두가 공감하시겠지만 Git은 이제 개발에 있어서 없어서는 안되는 필수기술로 자리잡았습니다.
그럼 궁금한 것이, 앞으로 Git의 운명은 어떻게 될까요?
지금과 같이 앞으로도 수십년간 VCS의 제왕의 자리에 올라있을까요? 아니면 BitKeeper때 그런 것처럼 새로운 기술에 의해 대체되는 운명을 맞을까요? 그건 모를 일인 것 같습니다!!!
